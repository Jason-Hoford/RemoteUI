# Known Limitations

## Unported APIs

### Bitmap Operations
- `addNamedBitmap()` — requires Android `Bitmap` object (platform-specific)
- `addBitmap(Object)` — same; no pure-Python equivalent
- 8 reference demos depend on these: texture, wake, bitmap-font, hostile-actor, confetti, etc.
- **Workaround**: Use `add_bitmap_data()` with raw pixel bytes if available

### Path from Text
- `buildPathFromText()` — converts text string to path using font metrics
- Requires font rasterization not available in pure Python
- 3 reference demos depend on this: indexing_demo, procedure_text_path_effects, paths_demos
- **Workaround**: None without adding a font rendering dependency (e.g., FreeType)

## Platform Differences

### Float Precision
Python uses 64-bit `float` internally (C `double`). All values are truncated
to 32-bit IEEE 754 at serialization time via `struct.pack('>f', value)`.
This can cause sub-ULP differences in complex expression chains, though none
have been observed in practice.

### Random Number Generation
`demo_maze` uses Python's `random` module which produces different sequences
than Java's `java.util.Random`. The maze structure differs but the binary
format is correct. Generated size: 11618 bytes vs reference: 5179 bytes.

### HashMap Iteration Order
Java `HashMap` iterates in bucket order, which depends on capacity and hash
function. The writer's `header_tag_order` parameter exists specifically to
match Java's iteration order for byte-identical output. Default order matches
the standard Java constructor with capacity 8.

## API Surface Gaps

### Internal-Only Java Methods Not Ported
- `checkAndClearForceSendingNewPaint()` — internal optimization
- Various internal accessors that are not part of the public creation API

### No Compose DSL
The Python port provides `RcContext` (matching Kotlin's `RemoteComposeContext`)
but does NOT provide Compose-level DSL (`RemoteRow`, `RemoteColumn`, etc.).
The context API is sufficient for all creation use cases.

## Performance

No performance optimization has been done. The generator is fast enough for
all current use cases (219 demos generate in ~2 seconds total). The `WireBuffer`
pre-allocates 1MB and grows as needed. For very large documents (>1MB), the
buffer doubling strategy may cause brief memory spikes.
