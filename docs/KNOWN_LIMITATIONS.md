# Known Limitations

## Bitmap Support

### What works
Bitmap data support is **fully implemented**. The wire protocol stores PNG-encoded
bytes, not Android `Bitmap` objects. Python provides multiple ways to add bitmaps:

- `add_bitmap_data(width, height, data)` — raw image bytes (TYPE_PNG_8888)
- `add_bitmap_png(width, height, png_data)` — PNG bytes with proper type encoding
- `add_bitmap_from_png(png_data, name=None)` — auto-extracts dimensions from PNG header
- `add_bitmap_from_file(path, name=None)` — loads PNG file from disk
- `add_named_bitmap(name, width, height, png_data)` — stores and names a bitmap
- `add_bitmap_url(url)` / `add_named_bitmap_url(name, url)` — URL references
- `add_bitmap_a8(width, height, data)` — alpha-8 format
- `create_bitmap(width, height)` — empty bitmap for render-to-texture
- All drawing operations: `draw_bitmap()`, `draw_scaled_bitmap()`, `draw_on_bitmap()`
- Bitmap fonts: `add_bitmap_font()`, `draw_bitmap_text_anchored()`, etc.

PNG dimensions are extracted from the standard IHDR header with no external
dependencies. Pillow is only needed for non-PNG image formats.

### What is not ported
The Java convenience methods `addBitmap(Object)` and `addNamedBitmap(name, Object)`
accept Android `Bitmap` objects. These specific signatures are not ported because
they are platform-specific wrappers — internally they just compress the Bitmap
to PNG bytes and call `storeBitmap()`. The Python equivalents (`add_bitmap_png()`,
`add_bitmap_from_png()`, `add_bitmap_from_file()`) accept PNG bytes directly,
which is what the wire format actually stores.

### Unmatched bitmap reference demos (8)
8 reference `.rc` files were generated by Kotlin demos that load bitmaps from
Android app resources (drawables, raw assets). These demos cannot be matched
because the original image data is embedded in the Android project, not available
as standalone files. The limitation is the missing image assets, not the Python
bitmap API.

Affected: `bitmap_font_watch`, `hostile_actor1`, `hostile_actor1_c`,
`impulse_demo_confetti_demo`, `texture_demo_*` (3), `wake_demo_wake_clock`.

## Unported APIs

### Path from Text
- `buildPathFromText()` — converts text string to path using font metrics
- Requires font rasterization not available in pure Python
- 3 reference demos depend on this: indexing_demo, procedure_text_path_effects, paths_demos
- **Workaround**: None without adding a font rendering dependency (e.g., FreeType)

These references were generated by `RemoteComposeContextAndroid` (Kotlin DSL)
which wraps content in Compose-level layout components. The Python port targets
the `RemoteComposeWriter` Java API. Porting these would require creating
individual demo scripts matching the exact Kotlin DSL calls.

Cannot be matched — the Kotlin demos load image data from Android app resources
that are not available as standalone files. See the Bitmap Support section above.

Cannot be matched without font rasterization.

## Platform Differences

### Float Precision
Python uses 64-bit `float` internally (C `double`). All values are truncated
to 32-bit IEEE 754 at serialization time via `struct.pack('>f', value)`.
This can cause sub-ULP differences in complex expression chains, though none
have been observed in practice.

### Random Number Generation
`demo_maze` uses Python's `random` module which produces different sequences
than Java's `java.util.Random`. The maze structure differs but the binary
format is correct. Generated size: 11618 bytes vs reference: 5179 bytes.

### HashMap Iteration Order
Java `HashMap` iterates in bucket order, which depends on capacity and hash
function. The writer's `header_tag_order` parameter exists specifically to
match Java's iteration order for byte-identical output. Default order matches
the standard Java constructor with capacity 8.

## API Surface Gaps

### Internal-Only Java Methods Not Ported
- `checkAndClearForceSendingNewPaint()` — internal optimization
- Various internal accessors that are not part of the public creation API

### No Compose DSL
The Python port provides `RcContext` (matching Kotlin's `RemoteComposeContext`)
but does NOT provide Compose-level DSL (`RemoteRow`, `RemoteColumn`, etc.).
The context API is sufficient for all creation use cases.

## Performance

No performance optimization has been done. The generator is fast enough for
all current use cases (219 demos generate in ~2 seconds total). The `WireBuffer`
pre-allocates 1MB and grows as needed. For very large documents (>1MB), the
buffer doubling strategy may cause brief memory spikes.
